# xBridge MCP — Development Workflow Standards (Optimized v4)

This is the single source of truth for all work on this project. Re-read and re-optimize at the start of every session.

## 1. Git Standards (Mandatory)
- Never work directly on main.
- Every piece of work starts on its own short-lived branch:
  - feat/<name> for new features
  - fix/<name> for bug fixes
  - chore/<name> for tooling, docs, standards
- Before any change: `git checkout -b <branch-name>`
- Commit early and often with conventional commits.
- After finishing a branch, merge or rebase cleanly to main.
- Parallel agents: each agent gets its own branch. Never share files between simultaneous agents.

## 2. Code Quality Standards
All code written for this project must be:
- Self-explanatory: clear variable names, small functions, obvious control flow. Comments only for non-obvious decisions.
- Reusable: prefer object-oriented modular design. Use classes for stateful components (tools, sessions, managers), protocols for interfaces, composition over inheritance. Extract common logic into reusable modules and base classes.
- Reliable: proper error handling, input validation, type hints (Python), tests for core paths. No silent failures.
- Modular boundaries: each class/module has one clear responsibility and can be imported and reused independently.

## 3. Development Standards (Clear Process)
- Small tasks (<1h): one agent does full cycle on one branch (research + implement + test + commit).
- Large tasks: research agent first (produces examples, tech choices, pitfalls, guidance in context file). Implementation agent follows.
- When creating any new tool, plugin, algorithm or utility: save code + immediately document in LIBRARY.md with purpose, API, examples, dependencies.
- Subagents: always use a shared .agent-context.md file. Every agent reads it on start and updates it on finish.
- Builders: treat claude-code CLI and grok-build CLI as normal workers/tools. Delegate the same way as any other agent.

## 4. Endpoint Classification Standards (Human vs Agent vs Internal)
Research from OpenAI, Anthropic, Stripe, Google and current MCP practices shows three distinct endpoint surfaces must be clearly separated:

- Human-user endpoints: CLI, web dashboards, natural language interfaces. Optimized for readability, progressive disclosure, error messages friendly to people.
- AI-agent-user endpoints: MCP tools, function schemas, structured JSON. Optimized for machine parsing, strict schemas, high-volume iterative calls, different rate limiting, intent verification. No natural language ambiguity.
- Internal modules: Pure library code, utilities, no network exposure. Reusable across the other two surfaces but never directly callable from outside.

Rules:
- Every new endpoint or tool must be labeled in code comments and docs as one of: HUMAN, AGENT, or INTERNAL.
- MCP tools (this project) are always AGENT surface.
- Never mix concerns in one module (e.g. an MCP handler must not contain human-facing HTML).
- Use separate files or clear module boundaries for each surface.
- Documentation (README, LIBRARY.md) must list which surface each component belongs to.
